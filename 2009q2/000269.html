On Fri, 24 Apr 2009 23:28:22 +0300
Eero Tamminen <eerot at users.berlios.de> wrote:

> >   - sound can be a problem and can generate latency ; hatari should
> > be run with sound off.
> 
> Laurent & Thomas got just DSP sound output working and that's also
> something that needs to be optimized later on, but in normal case it
> indeed makes sense to disable sound.

I think Laurent meant the fact that for sound output, the SDL starts a
second thread which could influence the performance of the main thread,
too. So for plain performance measurements, the sound should really be
disabled.

> > But one could imagine an option 'run for x VBLs then exit'. This
> > way, you could restore a memory snapshot of the same demo sequence
> > for example, run it full speed for 30*50 VBLs then exit and measure
> > the time it took.
> 
> Good idea, I added that to the patch (attached).

Nice idea, yes, looks good!

> > But the most precise way would certainly be to put some "probes" at
> > specific location when entering/exiting various functions and sum
> > the total time.
> 
> In that case it would be better just e.g. output it so that one can
> post-process it and calculate average/median/deviation to see how
> reliable the measurement is statistically.

Well, if you want to have "probes" for functions, you can also
simply compile Hatari with profiling support (add -pg to the CFLAGS and
use gprof to analyze it).
