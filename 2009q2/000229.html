On Sat, 4 Apr 2009, Eero Tamminen wrote:

> Hi,
>
> On Friday 03 April 2009, Thomas Huth wrote:
>>> I was looking at some other emulators having GUI debuggers and
>>> noticed that they show immediately something.   Would it be fine if
>>> Hatari CLI debugger would when invoked dump registers & mem also?
> [...]
>> Well, that could be helpful in some cases, but in some others, it also
>> might be just annoying. For example, I often only want to see the
>> disassembly at the current PC, so when the reg and memory dump
>> "pollutes" the screen that would just be a waste of space and time.
>>
>> And by the way, are you using the debugger at all? I just wonder,
>> since it does not make much sense to hex-dump the memory at the PC.
>> You normally rather want to see the disassembly of that region.
>
> Not yet, but I started to look into this DSP debugging thing Laurent
> suggested and though that dumping certain things always when invoking
> the debugger might be useful.
>
> I guess doing the other changes but leaving the calls to output functions
> commented out (and adding the DSP stubs) would be fine?
>
> One can then easily enable the line(s) for whather output he wants manually
> and re-compile Hatari.  Only annoying thing is that pressing the debugger
> key (Pause) again in Hatari window wouldn't continue the emulation after
> this info is dumped.  One needs to type "c" in console.

Well, I think that even if debug and pause use the same key, it's normal 
that when in debug mode pressing 'pause' again doesn't exit debug mode and 
simulate a 'c'.
Either we have a complete second window that would popup (similar to monst 
debugger for example) with some icons too that the user could use, or we 
just drop the user into the debug shell and he knows how to use it 
anyway.

What about a flag to tell what to do when we're in debug mode and the 
pause key is pressed ? For example --debug-start = none or dump_reg or 
dump_code or dump_all or ... ?

>
> Or maybe there should be a separate debug mode that would just dump the info
> and then continue emulation?  (then -D could take an argument for what it
> dumps whenever debugger key is pressed)
>

considering the speed at which the 68000 is changing its state, I don't 
see many interest in dumping the content of the register and continuing 
the emulation. I don't think anyone will succeed in dumping what he 
expects at the exact time.
