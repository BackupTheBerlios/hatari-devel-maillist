On Tuesday 06 October 2009, npomarede at corp.free.fr wrote:
> >> * For now binary prefix is "'".  I'll probably remove the '%' modulo
> >>   operation support so that binary numbers can use same prefix as rest
> >>   of the debugger.
>
> I think % would be better used as binary prefix, I'm not sure anyone uses
> modulo.

OK.


> >> * It doesn't support debugger number-base setting.  When I fix that,
> >> I'll probably remove support for octal and "ASCII" numbers as DebugUI
> >> doesn't support those.  Octal numbers can be useful only for BCD /
> >> nibble number calculations, but I couldn't come up with reasonable
> >> prefix for them. ;-)

Any suggestion for octal prefix?  What's used by Atari debuggers /
assemblers for that?


> >> * Calculations are done as doubles and signed longs.  Former means
> >> that divisions can have funny results when printed (due to conversion
> >> accuracy) and that shift and logical OR & AND operations support
> >> values only up to 0x7fffffff.  If floating point calculations aren't
> >> needed, I could change everything to use (signed) long longs?
>
> I never needed float point when doing asm in atari, I don't think we
> will need it in the debugger to fix some problems related to hatari.

Well, I was thinking that it could be useful when one is doing e.g. demo
coding (pre-calcs...), but I agree that this can probably be done
(better) "off-line" and it isn't that useful when debugging.
-> I'll change it to long longs.


> >> Comments?
>
> I'm not sure we should put too much extra evaluate commands in hatari ;
> of course it's your time, so you're free to do it, but in the end I think
> they're some much more complete calculators solutions that the user can
> run from another terminal if he really wants to (adding that to the fact
> that debugger mode is used by a very low percentage of users I guess)

Is there any free calculator that supports both binary & shift operations
and oct/dec/hex modes conveniently?  I haven't found any.

Btw. I nowadays use python interpreter to do calculations as it has in
recent years gained pretty good calculation support (order of predecense,
math functions, infinite precision, binary and shift operations etc) and is
available everywhere.  However, for bin/oct/hex it's still pretty
inconvenient:
	>>> hex(int(oct(20),8) + int(hex(10),16))
	0x1e
