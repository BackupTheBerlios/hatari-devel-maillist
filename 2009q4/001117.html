Uwe Seimet <Uwe.Seimet at seimet.de> wrote:

> Hi,
> 
> > Ok, I've changed it to long now, thanks for the hint.
> > Please note that this only works on 64-bit systems though. On 32-bit
> > systems, sizeof(long) is 4 bytes, so it is still limited to 2 GB.
> > I've searched for a better solution, but there does not seem to be a
> > really wide-spread portable way to get file sizes of huge files.
> 
> Under Linux stat or fstat might be an alternative, at least for
> determining the size hard disk image files. The image file size is
> always a multiple of 512 bytes, and fstat provides a blkcnt_t field
> which is the number of 512 byte blocks allocated by a file.

Yes, but according to the man page of fstat, st_blocks is not portable.
And according to the POSIX man page:

"The unit for the st_blocks member of the stat structure is not defined
 within IEEE Std 1003.1-2001. In some implementations it is 512 bytes.
 It may differ on a file system basis. There is no correlation between
 values of the st_blocks and st_blksize, and the f_bsize (from
 <sys/statvfs.h>) structure members."

(see the following link:
http://www.opengroup.org/onlinepubs/009695399/basedefs/sys/stat.h.html )

That doesn't sound very portable either.
