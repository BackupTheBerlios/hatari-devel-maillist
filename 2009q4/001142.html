Hi,

On Sunday 27 December 2009, Vincent Rivière wrote:
> In modern MiNT setups, we can use GDB to debug normal programs, but
> there is no tool for debugging special software like fVDI, XaAES or
> MyAES.

What specifically is the problem in debugging these?  Cannot Gdb  (run 
remotely e.g. over SSH) attach to multiple processes?  Or is the issue
finding out what these do also within MiNT (e.g. in XaAES kernel extension)?

I assume debugging these doesn't need MMU (Hatari can run MiNT,
but not with MP enabled as Hatari doesn't emulate MMU currently).


> If the debug info is in DRI format all the labels are in the executable, 
> its easy.

Do you have a pointer to DRI (on-disk) format documentation?


> If the debug info is in STABS format (the one produced by GCC), the
> symbols are in the executable, too,

Do you have any better documentation on the STABS on-disk data format
than e.g. this:
http://sourceware.org/gdb/download/onlinedocs/stabs.html#Symbol-Tables
?


> but the source-level information 
> require the sources, gdb is usually the only program able to exploit
> this. And Hatari should have some way to find the the sources.
>
> In the case of some additional debugging abilities would be added to
> Hatari, I think the display of the symbols only (no sources) would be
> enough at first.

I don't know anything (on any platform, not just Atari) that would allow
debugging all programs in a system, including kernel, at source level.  Gdb 
allows attaching to multiple processes at the same time (I think), but you
cannot debug both kernel, and processes kernel runs, at the same time.

The maximum level of support I was thinking of was being able to set
breakpoints to function and variable addresses and seeing these names when
doing disassembly or dumping memory contents.  Would that kind of a goal
be useful enough to be worth consider implementing during next year?


In the MiNT case I'm not sure how user could even specify properly the
process for which the debug symbols should be loaded for to the debugger.
How do you find out the process information with MiNT when only thing you
have is the HW memory contents?  (emulator doesn't have MiNT provided
information inside the emulation which Gdb can access)

Debugger could have some helpers which show information about the currently
loaded MiNT processes inside the emulation, including their memory
addresses, if there *is* a stable way to find that information from the
emulated memory somewhere.  This would be useful for reading the
dissassemby / trace output even before user could tell debugger from
which file to load debug symbols for given process/address.
