Hi,

On Monday 28 December 2009, Mickael Pointier wrote:
> I don't think such an emulator-integrated debugger should be able to
> understand all the file formats of the universe, but just being able to
> load symbols/address files would make it possible to break at any time,

Good idea!  This seems quite doable, I can add also TAB completion for
the symbol names.  I could add the symbol support first to (CPU) address
breakpoints.

Is the "nm" output format fine for the symbol/address information?

E.g. for Hatari it looks like this:
0804b750 T Audio_UnInit
0804b730 T Audio_Unlock
08186560 R Audio_fileid
081cee60 b AviFileHeader
081cee00 b AviParams
081ca684 D AviRecordDefaultCrop

I.e. address, symbols type (Text, Data, Bss) and symbol name.


> and have symbols in the rom, in the program, etc... if you add the
> possibility to add symbols from the debugger and then save them, you can
> progressively start to comment the program you are debugging.

What do you mean by "save them"?

Shouldn't it be enough just to be able to load the symbols?
And maybe remove them?

Syntax could look something like this:

* Add symbols from given file:
	addsymbols <offset> <filename>
* Remove symbols that fall within given range:
	delsymbols <address range>
* Show currently running TOS program (basepage) information
  (needed for the offset?):
	proginfo


> A real native debugger can also support non intrusive breakpoints,
> breakpoints on data changes, conditional breakpoints that do not hog the
> machine, etc....

Hatari debugger already supports these features.  I added them to v1.3 with
help from the others. (ideas mostly from Nicolas, and generic debugging
support from Thomas).
