On Thursday 03 December 2009, Eero Tamminen wrote:
> I've fixed that issue and added support for setting the file date&time.
...
> However, for some reason although Hatari sets the modification time
> succesfully even according to strace:
...
> and doesn't manipulate this file afterwards except by closing its FD,
> file system claims that its modification date is today:
...
> (Should utime() be called only after GEMDOS emu is asked to close the
> file?)

It was enough just to fflush() the FILE before setting its modification
time.  I forgot that FILE streams are buffered by the C-library.


Btw. Hopefully this works also on OSX and Windows, I'm not sure how portable
utime() function is.  Man page says "SVr4, POSIX.1-2001" for it and
"4.3BSD, POSIX.1-2001" for utimes() (OSX is BSDish), but states that
"POSIX.1-2001 marks utimes() legacy" so I used utime().
