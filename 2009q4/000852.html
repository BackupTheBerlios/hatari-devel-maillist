> > Truth to be told, I'm not so fond of it either (and don't use them in
> > my own projects[1]), but I think inconsistent naming conventions in
> > source code are even worse.  Renaming everything in current Hatari
> > code base not to have prefixes doesn't seem reasonable either.
>
> I second that. The worst thing is a mixture within one file.

But If one wants to use one's own naming convention within a file, at least
the global variables should follow naming consistent with other global
variables.  Otherwise the inconsistency spreads to multiple files.

(Within one's own file, one can then think the prefix as something that
differentiates the global variables from static or local ones.)


> Yes, and I've seen some projects already where people start mixing
> boolean and normal types, e.g. "a = 2" at one place, and then later "if
> (a == true)" ... guess what ... right, that's crappy code.

Btw. I guess you've seen also things like:
	if (0 == a)
?

That's very defensive programming. It avoids the mistake of accidentally
having an assignment (=) instead of (==) comparison (assignment lvalue
cannot be a constant, so build would fail).  However, it's not anymore
needed with GCC because GCC warns about assignments (used in conditions)
that aren't parenthesised.
