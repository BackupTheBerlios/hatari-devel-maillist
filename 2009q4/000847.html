Hi,

On Sunday 18 October 2009, npomarede at corp.free.fr wrote:
> > Btw. I changed AviRecording -> bRecordingAvi as other recording
> > bools used that name format.
>
> No please, I made this on purpose, I *really* don't like this microsoft
> naming scheme. The name should be self explanatory without having to add
> "sz", "b" or whatever microsoft once chose.
>
> I know hatari comes from a windows project at its basis, but no one is
> really prefixing/suffixing variables with their types nowaday in any
> project I know, so maybe we should not enforce this.

Truth to be told, I'm not so fond of it either (and don't use them in my own
projects[1]), but I think inconsistent naming conventions in source code are
even worse.  Renaming everything in current Hatari code base not to have
prefixes doesn't seem reasonable either.


Btw. another thing I haven't see elsewhere is prefixing also static
functions. And CamelCaps use differs from project to project, I've never
used it in my own projects.  In general it seems to be used more in C++
projects, maybe due to C++ being originally used more on Windows because
of MFC.

My own prefences match the Linux kernel coding guideline:
	http://www.kernel.org/doc/Documentation/CodingStyle

(Except for typedefs. In kernel there's a good reason not to hide structs
with typedefs because they can have performance overhead, I think in user
space readability matters more.)


> What is useful is the context in which the variable is used, and this
> should be enough IMO to see what a variable is for and what type it is
> (thing like "if (AviRecording==true)" are enough to see we're dealing
> with a boolean I think)

If you have a global function named AviRecording, then smallcaps b in
the beginning of the corresponding variable helps differentiating it.
Depending on context, compiler might not warn you about that.


> I know this is a matter of choice, but I really find this notation to be
> redundant. If some variables need to go from 16 bits int to 32, should we
> rename all prefixes from "w" to "dw" ? What matters most to understand
> the code is the stored value, not really the storage size.

If boolean changes to something else, I think the name of the variable
should indeed change (and not just whatever prefix it has). :-)
