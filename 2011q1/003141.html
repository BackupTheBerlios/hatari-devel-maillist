> What if I would pre-liminarily change --cpu-exact to --cycle-exact?

Regarding cpu options, I think we should really first have a working 
support, see what modifications it took to get there and then update/add 
manuals accordingly.

There's no need in rushing on doc for now, we don't know which cpu mode 
will remain in Hatari and how they might be customised to our use ; it 
will be the icing on the cake once everything else is working.


>
> And what if I would also shorten the --sound-buffer-size option to
> --sound-buffer?  Currently it and its description doesn't fit to 80 columns.
>
> (Unfortunately --sound-buffer-size was already in v1.4.0 release so it would
> be an API change.  I don't think it widely used though, anybody needing it
> should have set it to config file&  forgotten about it. :-))

I don't like the idea of changing existing options ; also 
"--sound-buffer" alone doesn't necessarily imply "size", it could me 
mono/stereo, or buffered versus unbuffered ?

Maybe it would be better to add a CR/LF in the description of 
--sound-buffer-size to have it on 2 lines ? (some options are already 
using 2 lines).

As for the historical 80 columns convention, I'm not sure it really 
applies nowaday. Do all terminals under kde/gnome/os x/... really open 
with 80 columns by default ? Is it really such a problem if lines are 
wider than 80 columns ?
In my case (and I think many people are doing the same), when I 
regularly use terminals, I always widen the window, 80 columns is just 
too short.

Best would be to have something similar to "man" where the text adapts 
to the width of the current shell (keeping indentation and so on), but 
that's kinda out of the scope of Hatari anyway.
