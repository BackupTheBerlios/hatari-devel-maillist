On sunnuntai 31 heinäkuu 2011, Nicolas Pomarède wrote:
> > With Hatari, in Hextracker the mouse moves _much_ slower in vertical
> > direction than on med-rez GEM desktop.  Does that happen also on real
> > ST?
>
> > ->  If yes, Hextracker doesn't scale mouse movements properly.
> 
> But even so, what causes the problem to appear for some people and not
> others?

Doesn't it?  Can you move the Hextracker mouse pointer to the lower part
of the screen in the windowed mode without grabbing the mouse (or installing
some Atari mouse accelerator, see below)?


> Is there a relation between the OS resolution and the mouse ?

HW-wise they're independent, but in med-rez there's more pixels to cover
on Atari side for the same amount of vertical mouse movements.  In GEM
the mouse at least seems to move across whole screen area vertically with
same amount of movement regardless of whether one uses low or med-rez.
Maybe GEM does some resolution dependent mouse movement scaling?

(Hatari also of course needs to compensate for its own screen zooming when
translating SDL events for host mouse movements to Atari mouse IKBD events.)


> Even if hextracker doesn't scale correctly, why does it work in
> fullscreen ? Is this because the screen X/Y ratio has a certain value ?

The reason is that unless mouse is grabbed (like happens with AltGr+M
or fullscreen), the mouse moves outside of the Hatari window and therefore
Hextracker doesn't anymore get relative mouse movement events.  And when
Hextracker scales the movements wrong, you cannot move pointer to everywhere
on the screen before host mouse pointer moves out of the Hatari window.


> If so, can't we compensate delta X / delta Y in Hatari when we detect
> the windowed resolution is not good ?

Only way to compensate for this Hextracker issue would be to make mouse
movement twice as slow for everything in med-rez.  I don't think
people would really appreciate it.

People typically install mouse accelerators to get faster mouse, I
don't understand why Hextracker specifically slows it down... :)

Note: you would get similar issue for any other mouse decelerator.

There isn't enough space for a text message and if user doesn't know

> At least this would allow the user to know in which state

I think this is a red herring.  During the last 8 years I've never
