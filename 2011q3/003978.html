On keskiviikko 03 elokuu 2011, Eero Tamminen wrote:
> Either user needs to use grab/fullscreen, or Hextracker needs
> to be fixed.   Personally I would prefer the latter, the mouse
> moves annoyingly slowly in it.

I found another program that has the same issue: Spectrum 512.

It would be useful if people could try some other med-rez programs
with mouse support to find out how wide spread this problem.
-> is it enough to mention the issue in compatibility list,
   or should some kludge added to Hatari for them?


> On sunnuntai 31 heinäkuu 2011, Eero Tamminen wrote:
> > I think that's because host mouse position compared to the Hatari
> > window position is different in fullscreen and in windowed mode and
> > therefore emulator gets (probably pretty large) mouse movement event
> > from SDL (which IKBD code splits into smaller -127 - 128 sized
> > movement packets).
> 
> I added some debug output to Hatari with the attached patch and
> this is what mouse movement events SDL reports...
> 
> 1. On pressing F11 to switch to FULLSCREEN
>    mouse: abs (840, 525) rel (424, 237)
> 
> 2. On pressing F11 to switch to WINDOWED mode
>    mouse: abs (412, 261) rel (-428, -264)
> 
> 3. On pressing F11 to switch to FULLSCREEN
>    mouse: abs (840, 525) rel (428, 264)
> 
> 4. On pressing F11 to switch to WINDOWED mode
>    mouse: abs (412, 261) rel (-428, -264)
> 
> Between F11 presses I didn't move the mouse.
> 
> "abs" is the absolute mouse co-ordinates reported by SDL
> and "rel" is the reported relative offset that gets given
> to IKBD (after it's scaled down by the zoom factor).
> 
> 
> As you can see, the mouse movements come from X11/SDL, Hatari
> just propagates them to the emulated program.

I tried storing the relative Hatari window mouse offset before toggling
fullscreen and restoring it afterwards, but it didn't help.  And making
it to apply in every case (with --desktop on/off, --desktop-st on/off etc)
would have required to copy same code to several different places.

Better if user just grabs mouse.
