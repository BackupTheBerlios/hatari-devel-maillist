On Tuesday 15 June 2010, Eero Tamminen wrote:
> Debugger catches the trap when PC points to the trap instruction, so
> it's not executed yet and we're not within a trap handler.  So I guess
> it's OK to use A7 directly also in this case to find what's in stack?

I commited the BIOS/XBIOS/GEMDOS/VDI/AES/Line-A/Line-F opcode
breakpoint support to the repo:
	http://hg.berlios.de/repos/hatari/rev/e31b534184f5

Please test and comment whether it works fine.

You could also comment on my opcode variable naming (on which value one can
set the breakpoints).  For example, to trace all VDI calls, use this:
	b  VdiOpcode ! VdiOpcode  &&  VdiOpcode < 0xffff  :trace

(0xffff is the default value for the opcode variables so you want to ignore
it when tracking the opcode value changes.  E.g. zero is valid opcode number
so that cannot be used as the default value.)

After comments (and getting back to net access in a week or couple)
I can then update the manual.
