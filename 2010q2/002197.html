On Monday 17 May 2010, Eero Tamminen wrote:
> > Something is indeed very much screwed in the OSX SDL 1.2 pipeline...
> > But unfortunately, performance is still lackluster (~40% cpu on 2.4Ghz
> > Core2Quad) with OpenGL blits. Atleast its possible to scale to any
> > arbitrary size with no cpu cost...
>
> Basically 3D HW / OpenGL helps with performance only in the cases where
> you're dealing with lots & complex geometry and where your texture
> content is static.  If all you use it for is pushing constantly updating
> texture content and basically no geometry,

Then suitable methods for this could be either the GL texture streaming[1]
or using some Video API.  SDL offers latter, but I don't know how common
its support is:
	http://sdl.beuc.net/sdl.wiki/SDL_Overlay

OSX SDL backend seems to support two of the formats according to its
internal QZ_CreateYUVOverlay() function code:
-------------
    if (format == SDL_YV12_OVERLAY ||
        format == SDL_IYUV_OVERLAY) {

        codec = kYUV420CodecType;
    }
    else {
        SDL_SetError ("Hardware: unsupported video format");
        return NULL;
    }
-----------


[1] http://www.opengl.org/registry/specs/ARB/pixel_buffer_object.txt
http://processors.wiki.ti.com/index.php/OpenGLES_Texture_Streaming_-_bc-cat_User_Guide
