> What about the global CurrentInstrCycles variable value?

It's not always correct ; it's more like a "basis", but some instructions
have a "dynamic" time (asr, lsr, movem, ...) that depend on the operand.
CurrentInstrCycles is useful for some classes of instruction often related 
to overscan timing, but it's sometimes a rough approximation.

In the end, "cycles = (*cpufunctbl[opcode])(opcode)" is the real value for 
the opcode.

> Or maybe use Cycles_GetCounter(CYCLES_COUNTER_CPU)?

CYCLES_COUNTER_CPU counts the number of cycles for the instruction + the 
time needed to process some interruption/exception that could occur at the 
same time, so it may be a little more that the time taken by instruction 
at a specific PC (also this counter is only set when DSP mode is enabled, 
as it's not needed for ST/STE emulation).

In your case, "cycles" above + nWaitStateCycles is really the time taken 
by an instruction (all overscan timings are based on this value).
