> Note that currently MHz value isn't changed at all because TOSes work with
> different MHz values.
>
> But if TOS version and other parameters don't match, wouldn't it be fair to
> assume that user did a mistake and just forgot to specify other parameters
> beside TOS?  If yes, then I think it would make sense to set also the MHz to
> most appropriate value for that TOS, right?
>
> I.e. if one gives e.g. a TT tos, it would be nice if Hatari would
> automatically set speed also for TT without needing to specify
> "--machine tt" or "--cpuclock 32" (or do similar thing in the options
> GUI)...

I agree that if the user choose a TOS version and we can identify this TOS 
and its cpu/blitter/video/whatever requirement, then it would be a good 
idea to force all the parameters (not only cpu type).

But do we have such a list ? Some tos versions can be compatible with 
several cpu/speed, in that case we should have a default setting.


Also, this should not prevent the user to specify "exotic" combinations 
when using the command line, for example "--tos tos_tt --cpuclock 16" 
should still be possible ; in that case we would read arguments from left 
to right, the rightmost argument would override the 32 Mhz implied by 
"--tos tos_tt"
